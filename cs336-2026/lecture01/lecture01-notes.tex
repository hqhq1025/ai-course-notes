\section{如何阅读这一讲}

CS336: \textit{Language Models From Scratch} 的第一讲同时承担三件事：解释为什么今天仍然值得从零构建语言模型；给出现代 language model 技术栈的历史和课程地图；正式进入第一个技术主题 tokenization。它不是一节只讲行政安排的导论课，而是用课程哲学把后续所有技术单元串起来。

\subsection{Spring 2026 的新版定位}

这已经是 CS336 第三次开课。新版继续保留 ``from scratch'' 的核心哲学，但讲义明确说会更重视 high value-per-time concepts：不要为了实现细节而丢掉整片森林。2026 版还会更多覆盖现代语言模型里的实际成分，例如 mixture of experts、long-context modeling 和 agents。

\begin{importantbox}{第一讲的主线}
这门课的核心问题不是“怎样调用一个强模型”，而是“给定数据、算力、显存和工程时间，怎样训练并理解一个尽可能好的语言模型”。第一讲从宏观产业背景出发，最后落到 tokenizer 的代码接口，正是为了展示 CS336 的学习路径：先建立全栈视野，再把每个模块拆成可实现、可度量、可优化的工程对象。
\end{importantbox}

\subsection{官方材料的形式：可执行讲义}

本讲的主要来源不是传统 PDF slides，而是一个 Python 程序 \texttt{lecture01-slides.py}。它通过 \texttt{text(...)}、\texttt{image(...)}、\texttt{link(...)} 等函数描述讲义内容，并把少量可运行代码嵌入叙事中。这个形式本身也是课程观点的一部分：现代语言模型课程不应只讲概念，还要让概念和代码处在同一份可检查的材料里。

\end{knowledgebox}

\subsection{本章小结}

第一讲不是普通导论，而是课程方法论的样板：从产业现实和研究抽象的断裂开始，解释为什么必须理解技术栈，再把课程五个单元和 tokenization 的第一步连接起来。后续学习时，应始终把每个细节放回“固定资源下最大化模型质量”的主问题中理解。

\section{为什么今天还要从零理解语言模型}

讲义给出的核心问题是：研究者正在越来越远离底层技术。2016 年左右，很多 NLP 研究者会自己实现模型、准备数据、训练并 debug。2018 年以后，BERT 等预训练模型让研究者更多下载模型再 fine-tune。今天，许多研究甚至直接变成 prompt GPT、Claude、Gemini 等 API 模型。

\subsection{抽象层上移带来的生产力与风险}

抽象层上移当然提高了生产力。调用 API 比自己训练模型快得多，fine-tune 一个预训练模型也比从零训练可靠得多。但语言模型 API 的抽象并不像成熟编程语言或操作系统那样稳定封装。它们会在很多地方“漏出来”：

\begin{itemize}
    \item 模型上下文长度、tokenizer、system prompt 和 decoding 设置会改变任务行为。
    \item 训练数据和数据过滤策略会影响知识覆盖、偏见、记忆和评测污染。
    \item 后训练和安全策略会改变模型愿不愿意回答、怎样回答。
    \item 推理系统的 KV cache、batching、路由和工具调用会影响延迟、成本和可用能力。
    \item 闭源模型的版本更新可能让同一个 prompt 的行为发生漂移。
\end{itemize}

\begin{warningbox}{只会使用 API 的局限}
如果研究者只看到 API 表面，就很难判断失败来自哪里：是任务定义不清、prompt 不稳、tokenizer 切分异常、训练数据缺失、模型容量不足，还是推理系统约束。CS336 的立场是：要做 fundamental research，就必须能撕开抽象层，理解模型和系统的内部机制。
\end{warningbox}

\subsection{从 2016 到今天：研究方式的迁移}

\begin{table}[H]
\centering
\begin{tabular}{p{2.3cm}p{4.4cm}p{6.2cm}}
\toprule
\textbf{阶段} & \textbf{常见工作方式} & \textbf{研究者容易掌握的知识} \\
\midrule
2016 前后 & 自己实现并训练模型 & 数据处理、训练稳定性、模型结构、loss 曲线、硬件瓶颈 \\
2018 前后 & 下载 BERT 等模型并 fine-tune & 下游任务适配、超参数调节、预训练表示的迁移性 \\
今天 & prompt 或调用闭源 API & 交互设计、prompt 工程、应用封装，但底层机制不透明 \\
\bottomrule
\end{tabular}
\caption{语言模型研究中抽象层级的变化。}
\end{table}

课程并不是否定抽象。好的抽象是技术进步的前提。但当抽象仍在快速变化、底层机制会直接影响科学结论时，研究者不能只停留在最高层。CS336 的哲学是 understanding via building：通过亲手构建 tokenizer、Transformer、optimizer、training loop、systems 和 alignment 组件，获得能跨模型版本迁移的判断力。

\begin{importantbox}{从零构建不是怀旧}
``From scratch'' 不是为了重复工业界已经做过的大规模训练，而是为了重新获得因果理解：当 loss 降不下去、GPU 利用率很低、perplexity 异常、模型忽然记不住长上下文时，你知道应该检查哪个模块、用什么指标、怎样设计下一次实验。
\end{importantbox}

\subsection{本章小结}

今天的语言模型抽象层很高，但仍然高度 leaky。CS336 认为，只会调用强模型不足以支撑基础研究；真正可迁移的能力来自对技术栈内部机制的理解。第一讲因此把“从零构建”作为研究训练，而不是把它当成生产环境复刻。

\section{工业化语言模型与课堂可达性的边界}

现代 frontier model 已经高度工业化。讲义举了两个尺度感很强的例子：GPT-4 据称训练成本约为 100M USD；xAI 在 2025 年为 Grok 训练建设了 230K GPU 规模的集群。即使这些数字有不确定性，它们仍然说明一个事实：前沿模型的生产方式已经超出普通课程、实验室和个人可复现范围。

\subsection{闭源报告与不可复现细节}

讲义引用 GPT-4 technical report 的片段，强调前沿模型不会公开足够细节。模型规模、训练数据、训练步数、优化细节、系统并行策略、后训练流程都可能被省略或模糊处理。

\begin{warningbox}{不能把小模型当作前沿模型的缩小版}
课堂可以训练小于 1B 参数的模型，但小模型不一定代表大模型。随着参数量、上下文长度、batch size 和数据规模变化，FLOPs 分布、瓶颈位置、能力曲线和调参直觉都可能改变。因此，CS336 会区分“可迁移的机制”与“可能只在某个尺度成立的经验直觉”。
\end{warningbox}

讲义用 attention 与 MLP 的 FLOPs 占比变化提醒学生：即使架构名字仍叫 Transformer，实际计算画像也会随规模变化。某些配置下 attention 更重要，某些配置下 MLP 或通信更重要；长上下文又会进一步改变 prefill 和 decode 的成本结构。

\end{knowledgebox}

\subsection{Bitter Lesson 的正确读法}

讲义把 ``The Bitter Lesson'' 的常见误读拆开：错误读法是“规模就是一切，算法不重要”；正确读法是“能随着资源增长而持续受益的算法最重要”。第一讲把它压缩为一个简单公式：

$$
\text{accuracy} = \text{efficiency} \times \text{resources}
$$

\begin{itemize}
    \item \(\text{accuracy}\)：模型最终效果，可以是 loss、perplexity、benchmark 表现或真实任务质量。
    \item \(\text{efficiency}\)：单位资源转化为有效能力的效率，包括算法效率、系统效率、数据效率和调参效率。
    \item \(\text{resources}\)：可用数据、训练 FLOPs、显存、通信带宽、工程时间和推理预算。
\end{itemize}

这个公式不是严格物理定律，而是课程的设计原则。当前沿资源不可得时，课堂最应该训练的是 efficiency：同样的 GPU、同样的数据、同样的时间，谁能得到更好的 loss、更高的吞吐、更稳定的训练。

\begin{importantbox}{Bitter Lesson 对 CS336 的意义}
大规模并不让算法失去价值，反而放大了效率的价值。小模型阶段浪费一点算力只是慢几分钟；前沿模型阶段同样比例的浪费可能意味着数百万美元和数周排队时间。CS336 后续所有单元都可以看成对 efficiency 的不同分解。
\end{importantbox}

\subsection{本章小结}

前沿模型不可在课堂复现，但课堂仍然能学习可迁移的 mechanics 和 mindset。尺度会改变系统瓶颈和能力曲线，因此课程会谨慎处理 intuition。第一讲用 ``accuracy = efficiency x resources'' 把全课核心目标固定下来：在资源约束下最大化模型质量。

\section{语言模型发展版图}

第一讲给出一条从 pre-neural 到 agents 的历史线索。它不是为了背年份，而是为了说明今天的语言模型技术栈如何由许多独立进展叠加而成。

\subsection{Pre-neural：熵、n-gram 与传统系统}

早期 language model 主要服务于英语熵估计、机器翻译和语音识别。Shannon 的语言熵工作说明了“预测下一个符号”本身就是理解语言统计结构的基本方式。N-gram 模型则把上下文截断到固定长度，用计数和 smoothing 做概率估计。

从今天看，n-gram 最大的问题是无法学习长距离依赖，也无法共享语义相近片段的统计强度。但它留下了两个核心观念：第一，语言建模可以形式化为序列概率；第二，模型质量可以用 held-out likelihood 或 perplexity 衡量。

\subsection{Neural ingredients：2010 年代的关键积木}

2010 年代逐步出现了现代语言模型的关键积木：

\begin{itemize}
    \item LSTM 让模型能通过门控状态处理较长序列。
    \item 第一批 neural language model 用分布式表示缓解 n-gram 稀疏性。
    \item Seq2seq 和 attention 推动机器翻译从 feature engineering 转向端到端学习。
    \item Adam 优化器降低了大规模神经网络训练的调参难度。
    \item Transformer 用 self-attention 和高度并行的结构重塑了序列建模。
    \item Mixture of Experts、GPipe、ZeRO、Megatron-LM 等工作把模型扩展推向系统层。
\end{itemize}

\begin{knowledgebox}{Transformer 不是孤立发明}
Transformer 的成功不是因为一个模块突然替代一切，而是因为它把 attention、并行训练、深层网络优化和硬件友好计算组合到一起。CS336 后续讲架构时，会把这些“积木”拆开，而不是只记住一个 encoder-decoder 图。
\end{knowledgebox}

\subsection{Early foundation models 与 scaling}

ELMo、BERT 和 T5 展示了预训练再适配的能力。随后 GPT-2、GPT-3、PaLM、Chinchilla 等模型把 scaling law 推到中心位置。这里的关键转变是：模型开发不再只是“试一个架构”，而是要用小规模实验预测大规模训练结果，用 compute budget 决定参数量、token 数和调参策略。

\begin{table}[H]
\centering
\begin{tabular}{p{3cm}p{5.8cm}p{5.2cm}}
\toprule
\textbf{阶段} & \textbf{代表材料} & \textbf{给 CS336 的启发} \\
\midrule
早期预训练 & ELMo、BERT、T5 & 预训练表示可以迁移，下游任务不必从零学习语言结构。 \\
拥抱规模 & GPT-2、GPT-3、PaLM & 规模带来泛化和 in-context learning，但也带来资源规划问题。 \\

\subsection{开放模型：从 open-weight 到 open development}

\begin{importantbox}{开放模型为什么对教学重要}

\end{importantbox}

\end{importantbox}

系统单元的目标是 squeeze the most out of the hardware。讲义强调 resource accounting：模型参数必须从 HBM 移到 SM 做计算，kernel launch、memory bandwidth、communication bandwidth 都会限制吞吐。

\subsection{Scaling laws：用小规模实验规划大规模训练}

Scaling 单元让学生学习如何用小实验预测大训练。核心公式之一是训练 FLOPs 的粗略估算：

$$
C \approx 6ND
$$

\begin{itemize}
    \item \(C\)：训练计算量，单位通常为 FLOPs。
    \item \(N\)：模型参数量。
    \item \(D\)：训练 token 数。
    \item 系数 \(6\)：来自 Transformer 训练中前向和反向计算的常用近似，具体值会随实现和架构细节变化。
\end{itemize}

这个公式让一个抽象问题变成可规划问题：给定 \(C\)，应该增大 \(N\)，还是增加 \(D\)？经典 compute-optimal scaling laws 会通过 isoflop curves 在多个小预算下寻找最优参数量，再外推到更大预算。

\begin{figure}[H]
\centering
\includegraphics[width=0.95\textwidth]{chinchilla-isoflop.png}
\caption{Chinchilla 风格 isoflop 分析：在固定 FLOPs 下寻找参数量与 token 数的平衡。}
\figsource{Stanford CS336 Spring 2026 官方可执行讲义 \texttt{lecture01-slides.py}，图片 \texttt{images/chinchilla-isoflop.png}。}
\end{figure}

讲义给出的简化 takeaway 是 \(D \approx 20N\)，例如 70B 参数模型大约对应 1.4T tokens。但它也提醒：这个结论没有直接考虑 inference cost。如果部署成本重要，可能更偏好较小模型训练更多 token。

\subsection{Data：数据处理是模型质量的一部分}

数据单元从一个更根本的问题开始：我们希望模型具备什么能力？多语言能力、对话能力、代码能力、agentic coding 能力，对数据来源和数据处理提出完全不同的要求。

讲义把 data pipeline 拆成 evaluation、curation、transformation、filtering、deduplication、data mixing、rewriting/synthetic data 等环节。每一环都影响 efficiency：低质量数据会浪费训练 FLOPs；重复数据会增加 memorization 风险；错误混合比例会让模型在关键能力上欠训练。

\begin{knowledgebox}{数据不是“更多就好”}
在资源约束下，数据质量和数据匹配度会直接影响训练效率。一个 token 既可以来自高质量代码，也可以来自乱码网页；它们消耗相近 FLOPs，却带来完全不同的学习信号。CS336 把数据处理放在独立单元，是因为它已经成为语言模型工程的核心部分。
\end{knowledgebox}

\subsection{Alignment：从 next-token prediction 到弱监督优化}

Alignment 单元从一个事实出发：预训练模型学会了 next-token prediction，但真实系统还需要遵循指令、拒绝不安全请求、调用工具、完成长程任务。讲义给出的基本模板是：

\begin{enumerate}
    \item 从模型生成 responses。
    \item 用 human、verifier 或 LM judge 给 responses 打分。
    \item 更新模型，让它更偏好高分 responses。
\end{enumerate}

课程会涉及 PPO、DPO 和 GRPO。讲义特别提醒，RL algorithms 在大规模系统中不只是数学公式，还要求高吞吐 inference、异步 rollouts、on-policyness 与系统效率之间的权衡。

\subsection{本章小结}

五个单元构成一条完整语言模型生产线：basics 给出模型和训练核心；systems 让它跑得快；scaling laws 让大规模实验可预测；data 决定学习信号质量；alignment 把预训练能力转向真实交互。所有单元都服务于同一个问题：在固定资源下训练和部署最好的模型。

\subsection{Tokenization 也是 efficiency 决策}

Tokenization 常被当成预处理，但第一讲把它放进 efficiency lens。原因很直接：Transformer 的 attention 成本随序列长度增长很快。如果 tokenizer 把同样文本切成更多 token，模型每一步都要处理更长序列，训练和推理都变贵。

一个粗略的 self-attention 复杂度可以写成：

$$
\text{attention cost} = O(n^2 d)
$$

\begin{itemize}
    \item \(n\)：token 序列长度。
    \item \(d\)：hidden dimension。
    \item \(O(n^2 d)\)：表示每个 token 需要与其他 token 交互，序列长度翻倍会显著增加注意力计算。
\end{itemize}

如果 byte tokenizer 把 1000 bytes 变成约 1000 tokens，而 BPE 把它压成约 250 tokens，那么 attention 相关计算可能相差一个数量级。这个估算不等于完整训练成本，但足以说明 tokenizer 对系统效率的影响。

\begin{warningbox}{高压缩率不一定越高越好}
增加 vocabulary size 可以提高 bytes per token，但也会带来稀疏性：每个 token 的训练样本更少，embedding 和输出层更大，罕见 token 更难学。好的 tokenizer 不是简单追求最短序列，而是在覆盖性、压缩率、词表大小和学习难度之间折中。
\end{warningbox}

\subsection{本章小结}

Efficiency 是 CS336 的统一语言。Tokenization 决定序列长度，systems 决定硬件利用率，scaling laws 决定预算分配，data 决定每个 token 的学习价值，alignment 决定后训练反馈是否高效。第一讲提前建立这套语言，是为了让后续每个模块都能被放回同一个资源约束框架中。

\section{Tokenization 的基本定义}

Lecture 1 的第一个正式技术单元是 tokenization。原始文本通常表示为 Unicode string，而语言模型在整数 token 序列上建模。因此需要一个 tokenizer 在字符串和 token IDs 之间转换。

\subsection{Tokenizer 抽象接口}

官方讲义定义了一个非常小的抽象类：

\begin{lstlisting}[caption={Tokenizer 的抽象接口}]
class Tokenizer(ABC):
    """Abstract interface for a tokenizer."""
    def encode(self, string: str) -> list[int]:
        raise NotImplementedError

    def decode(self, indices: list[int]) -> str:
        raise NotImplementedError
\end{lstlisting}

这个接口虽小，但包含两个必须满足的要求：

\begin{itemize}
    \item \texttt{encode}：把原始字符串转成整数序列，让模型可以查 embedding 并进行序列建模。
    \item \texttt{decode}：把整数序列还原成字符串，让生成结果能回到人类可读文本。
\end{itemize}

\begin{importantbox}{Tokenizer 的本质}
Tokenizer 是语言模型的第一层表示学习前处理。它决定模型的“原子单位”：模型不是直接看到字符、byte 或词，而是看到 tokenizer 选择出来的 token IDs。这个选择会影响上下文长度、词表大小、训练稳定性、生成质量和系统成本。
\end{importantbox}

\subsection{Round-trip：可逆性是底线}

一个 practical tokenizer 至少应该满足 round-trip：

$$
\texttt{decode}(\texttt{encode}(x)) = x
$$

\begin{itemize}
    \item \(x\)：原始输入字符串。
    \item \(\texttt{encode}(x)\)：token IDs。
    \item \(\texttt{decode}(\cdot)\)：把 IDs 映射回字符串的过程。
\end{itemize}

如果 round-trip 不成立，模型训练时看到的序列与最终生成文本之间就会出现不可控信息损失。某些特殊 token 或 normalization 规则可能有例外，但主路径必须可解释、可测试。

\subsection{Compression ratio：bytes per token}

讲义用 compression ratio 衡量 tokenization 的序列效率：

$$
\text{compression ratio} =
\frac{\text{number of UTF-8 bytes}}{\text{number of tokens}}
$$

\begin{itemize}
    \item 分子是原始字符串按 UTF-8 编码后的 byte 数。
    \item 分母是 tokenizer 输出的 token 数。
    \item 值越大，表示同样文本被表示成更短 token 序列。
\end{itemize}

例如字符串 \texttt{Hello,} 加上中文字符会包含 ASCII byte 和多 byte Unicode 字符。character tokenizer 可能 token 数较少但词表巨大，byte tokenizer 词表很小但 compression ratio 固定为 1，BPE 则试图用数据驱动方式合并常见 byte 片段。

\subsection{GPT-5 tokenizer 的观察}

官方讲义用 OpenAI \texttt{tiktoken} 的 \texttt{o200k\_base} 编码器作为例子，并让学生观察真实 tokenizer 的行为：

\begin{itemize}
    \item 一个词和它前面的空格常常属于同一个 token，例如 \texttt{" world"}。
    \item 同一个词在句首和句中可能对应不同 token，例如 \texttt{"hello hello"}。
    \item 数字通常每隔几位切开，而不是把整个长数字当成一个 token。
    \item vocabulary size 越大，潜在 compression ratio 越高，但 token 统计也越稀疏。
\end{itemize}

\begin{knowledgebox}{为什么空格会进 token}
现代 BPE/byte-level BPE tokenizer 常把空格和后续词片段合在一起，因为英文中“空格 + 单词”是高频模式。这样做提高压缩率，也让模型更容易区分词首、词中和句首位置。但它也说明 token 并不等于人类直觉中的 word。
\end{knowledgebox}

\subsection{本章小结}

Tokenization 把 Unicode 字符串变成模型可处理的整数序列。一个 tokenizer 至少要可逆，同时要在 compression ratio、vocabulary size 和学习难度之间折中。第一讲接下来用 character、byte、word 三种朴素方案说明为什么需要 BPE。

\section{三种朴素 tokenizer 及其失败方式}

在进入 BPE 之前，讲义先让学生实现 character tokenizer、byte tokenizer 和 word tokenizer。这个顺序很重要：只有看清朴素方案的失败方式，才能理解 BPE 为什么是一个实用折中。

\subsection{Character tokenizer：直观但词表尴尬}

Character tokenizer 把字符串看成 Unicode code points。Python 中可以用 \texttt{ord} 把字符转成整数，用 \texttt{chr} 转回来。

\begin{lstlisting}[caption={CharacterTokenizer：用 Unicode code point 表示字符}]
class CharacterTokenizer(Tokenizer):
    """Represent a string as a sequence of Unicode code points."""
    def encode(self, string: str) -> list[int]:
        return list(map(ord, string))

    def decode(self, indices: list[int]) -> str:
        return "".join(map(chr, indices))
\end{lstlisting}

它的优点是概念直接、round-trip 容易测试。问题是 Unicode 字符空间很大，约有 150K 级别字符；同时很多字符极其罕见，模型需要为稀有字符保留 embedding 和输出概率，却很少得到训练样本。

\subsection{Byte tokenizer：覆盖性强但序列太长}

Byte tokenizer 把字符串先编码成 UTF-8 bytes，每个 byte 是 0 到 255 的整数。

\begin{lstlisting}[caption={ByteTokenizer：先转 UTF-8 bytes，再把 byte 当 token}]
class ByteTokenizer(Tokenizer):
    """Represent a string as a sequence of bytes."""
    def encode(self, string: str) -> list[int]:
        string_bytes = string.encode("utf-8")
        indices = list(map(int, string_bytes))
        return indices

    def decode(self, indices: list[int]) -> str:
        string_bytes = bytes(indices)
        string = string_bytes.decode("utf-8")
        return string
\end{lstlisting}

Byte tokenizer 的词表非常干净：只需要 256 个基础值，且能覆盖所有 Unicode 文本。问题是 compression ratio 固定为 1 byte/token。对 Transformer 来说，这会让序列过长，attention 和 KV cache 成本都变高。

\begin{table}[H]
\centering
\begin{tabular}{p{3.4cm}p{4.3cm}p{5.9cm}}
\toprule
\textbf{方案} & \textbf{优势} & \textbf{主要问题} \\
\midrule
Character tokenizer & 与人类字符直觉接近，容易可逆 & Unicode 空间大，稀有字符多，词表利用率低 \\
Byte tokenizer & 词表固定为 256，覆盖所有 UTF-8 文本 & 序列太长，compression ratio 低，attention 成本高 \\
Word tokenizer & token 语义强，压缩率通常较好 & 词表无界，稀有词和未登录词处理困难 \\
\bottomrule
\end{tabular}
\caption{三种朴素 tokenizer 的核心取舍。}
\end{table}

\subsection{Word tokenizer：语义自然但开放词表难处理}

Word tokenizer 更接近传统 NLP：用规则把字符串切成词和标点。讲义示例使用正则规则，让连续字母数字组成词，其余字符单独保留。

\begin{lstlisting}[caption={朴素 word tokenizer 的切分思路}]
string = "I'll say supercalifragilisticexpialidocious!"
chunks = regex.findall(r"\w+|.", string)
# 可能得到: ["I", "'", "ll", " ", "say", " ", "supercalifragilisticexpialidocious", "!"]
\end{lstlisting}

它的问题来自 open vocabulary。训练集中没见过的新词怎么办？如果统一映射为 \texttt{UNK}，就丢失了拼写信息，也会污染 perplexity 计算。如果把所有词都加入词表，词表会巨大且长尾严重。

\begin{warningbox}{UNK token 的代价}
UNK 看似解决了未登录词，实际上把所有未知词压成同一个符号。模型无法区分一个新的人名、一个新代码标识符、一个拼写错误和一个专业术语。对开放域语言模型来说，这种信息损失很难接受。
\end{warningbox}

\subsection{一个具体例子：三种方案的数值直觉}

考虑字符串 \texttt{Hello,}、一个地球 emoji、中文 \texttt{你好} 和感叹号的混合文本。按 UTF-8 编码，ASCII 字符通常占 1 byte，中文字符通常占 3 bytes，emoji 可能占 4 bytes。于是：

\begin{itemize}
    \item character tokenizer 的 token 数接近人类看到的字符数，但词表必须覆盖很大的 Unicode 空间。
    \item byte tokenizer 的 token 数等于 UTF-8 bytes 数，覆盖性完美但序列最长。
    \item word tokenizer 对英文词可能压缩良好，但对中文、emoji、代码和未见词需要额外规则。
\end{itemize}

这个例子说明，tokenizer 不是纯语言学问题，而是表示效率问题。语言模型需要一个既不丢信息、又不让序列爆炸、还不让词表爆炸的中间方案。

\subsection{本章小结}

Character tokenizer、byte tokenizer 和 word tokenizer 分别占据三个极端：字符直观但词表大，byte 覆盖性强但序列长，word 语义自然但开放词表困难。BPE 的动机正是把 byte 的覆盖性和 chunk 的压缩效率结合起来。

\section{Byte Pair Encoding：从压缩算法到语言模型 tokenizer}

BPE 最初由 Philip Gage 在 1994 年作为数据压缩算法提出，后来被 Sennrich 等人引入 neural machine translation，又被 GPT-2 等模型采用。它的核心思想很简单：从 byte 级 token 开始，反复合并最常见的相邻 token pair。

\subsection{BPE 的基本数据结构}

官方讲义把 BPE 参数写成一个 dataclass：

\begin{lstlisting}[caption={BPE tokenizer 需要的两张表}]
@dataclass(frozen=True)
class BPETokenizerParams:
    """All you need to specify a BPETokenizer."""
    vocab: dict[int, bytes]              # index -> bytes
    merges: dict[tuple[int, int], int]   # index1,index2 -> new_index
\end{lstlisting}

\begin{itemize}
    \item \texttt{vocab} 告诉 decode：每个 token ID 对应哪段 bytes。
    \item \texttt{merges} 告诉 encode：哪些相邻 token pair 应该被合并成新 token。
\end{itemize}

初始时，\texttt{vocab} 包含 256 个 byte token。每做一次 merge，就新增一个 token ID，例如 256、257、258，并把它对应到两个旧 token 的 byte 拼接。

\begin{importantbox}{BPE 的关键折中}
BPE 保留 byte-level fallback，所以不会遇到真正无法表示的字符串；同时它把高频 byte 序列合成更长 chunk，提高 compression ratio。它不是最优分词算法，但在覆盖性、效率和实现复杂度之间取得了非常实用的平衡。
\end{importantbox}

\subsection{merge 函数的细节}

讲义给出一个简单但重要的 \texttt{merge} 函数：

\begin{lstlisting}[caption={把所有目标 pair 替换为新 token}]
def merge(indices: list[int], pair: tuple[int, int], new_index: int) -> list[int]:
    """Return `indices`, but with all instances of `pair` replaced with `new_index`."""
    new_indices = []
    i = 0
    while i < len(indices):
        if i + 1 < len(indices) and indices[i] == pair[0] and indices[i + 1] == pair[1]:
            new_indices.append(new_index)
            i += 2
        else:
            new_indices.append(indices[i])
            i += 1
    return new_indices
\end{lstlisting}

这个循环里最容易忽略的是 \texttt{i += 2}。一旦匹配到 pair，两个旧 token 被一个新 token 替代，扫描位置必须跳过两个元素，避免重叠合并。例如序列 \([a,a,a]\) 合并 pair \((a,a)\) 时，结果应是 \([aa,a]\)，而不是不受控地同时合并第 1-2 和第 2-3 个 token。

\begin{warningbox}{BPE merge 的 off-by-one 风险}
实现 BPE 时，很多 bug 不来自算法概念，而来自扫描边界：最后一个 token 没有右邻居；匹配后要跳过两个位置；不匹配时只前进一步；pair 的合并顺序必须保持训练时的顺序。Assignment 1 的单元测试通常会抓这些细节。
\end{warningbox}

\subsection{训练 BPE：统计相邻 pair}

训练过程如下：

\begin{enumerate}
    \item 把训练字符串编码成 UTF-8 bytes。
    \item 统计当前 token 序列中所有相邻 pair 的出现次数。
    \item 找出出现次数最多的 pair。
    \item 给它分配一个新 token ID，并更新 \texttt{vocab} 和 \texttt{merges}。
    \item 调用 \texttt{merge} 替换序列中的所有该 pair。
    \item 重复指定次数。
\end{enumerate}

\begin{lstlisting}[caption={教学版 BPE 训练循环}]
def train_bpe(string: str, num_merges: int) -> BPETokenizerParams:
    indices = list(map(int, string.encode("utf-8")))
    merges: dict[tuple[int, int], int] = {}
    vocab: dict[int, bytes] = {x: bytes([x]) for x in range(256)}

    for i in range(num_merges):
        counts = count_adjacent_pairs(indices)
        pair = max(counts, key=counts.get)

        new_index = 256 + i
        merges[pair] = new_index
        vocab[new_index] = vocab[pair[0]] + vocab[pair[1]]
        indices = merge(indices, pair, new_index)

    return BPETokenizerParams(vocab=vocab, merges=merges)
\end{lstlisting}

\subsection{Worked example：\texttt{the cat in the hat}}

讲义用 \texttt{the cat in the hat} 训练 3 次 merge。初始 byte 序列对应字符：

\[
\texttt{t h e \textvisiblespace c a t \textvisiblespace i n \textvisiblespace t h e \textvisiblespace h a t}
\]

每次选择最高频相邻 pair，并把它变成新 token：

\begin{table}[H]
\centering
\begin{tabular}{p{1.2cm}p{3cm}p{2.2cm}p{6.4cm}}
\toprule
\textbf{轮次} & \textbf{被合并 pair} & \textbf{新 token} & \textbf{合并后的 token 片段} \\
\midrule
1 & \texttt{t} + \texttt{h} & 256 = \texttt{th} & \texttt{th e \textvisiblespace c a t \textvisiblespace i n \textvisiblespace th e \textvisiblespace h a t} \\
2 & \texttt{th} + \texttt{e} & 257 = \texttt{the} & \texttt{the \textvisiblespace c a t \textvisiblespace i n \textvisiblespace the \textvisiblespace h a t} \\
3 & \texttt{the} + space & 258 = \texttt{the\textvisiblespace} & \texttt{the\textvisiblespace c a t \textvisiblespace i n \textvisiblespace the\textvisiblespace h a t} \\
\bottomrule
\end{tabular}
\caption{BPE 在短字符串上的三次 merge 示例。}
\end{table}

这个例子有两点值得注意。第一，BPE 不知道英文语法，只看相邻 token pair 的频率；\texttt{the} 和后面的空格合并，是因为 \texttt{the\textvisiblespace} 在语料中高频。第二，merge 后的 token 不一定等于完整单词，也可能是前缀、后缀、空格加词、词的一部分或跨脚本片段。

\subsection{使用 BPE tokenizer}

训练得到 \texttt{params} 后，编码新字符串时先转成 bytes，再按 \texttt{merges} 的顺序应用合并：

\begin{lstlisting}[caption={教学版 BPE encode/decode}]
class BPETokenizer(Tokenizer):
    """BPE tokenizer given a set of merges and a vocabulary."""
    def __init__(self, params: BPETokenizerParams):
        self.params = params

    def encode(self, string: str) -> list[int]:
        indices = list(map(int, string.encode("utf-8")))
        # Note: this is a very slow implementation
        for pair, new_index in self.params.merges.items():
            indices = merge(indices, pair, new_index)
        return indices

    def decode(self, indices: list[int]) -> str:
        bytes_list = list(map(self.params.vocab.get, indices))
        string = b"".join(bytes_list).decode("utf-8")
        return string
\end{lstlisting}

这里的 \texttt{encode} 很慢，因为它对每个输入都遍历所有 merges。真实 tokenizer 会用更高效的数据结构，只处理当前序列中可能发生的合并。

\begin{knowledgebox}{BPE 是“训练出来的 tokenizer”}
BPE 的 vocabulary 和 merges 来自训练语料，而不是人工语言学词典。因此，不同语料、不同特殊 token 策略、不同词表大小会产生不同 tokenization。模型的 tokenizer 一旦固定，就成为模型接口的一部分，不能随意替换。
\end{knowledgebox}

\subsection{本章小结}

BPE 从 byte 级表示出发，反复合并高频相邻 pair，得到一个既可覆盖任意文本、又能压缩常见片段的 tokenizer。理解 BPE 的关键不是背算法名字，而是理解 \texttt{vocab}、\texttt{merges}、merge 顺序和 round-trip 如何共同保证编码与解码。

\section{Assignment 1 中 tokenizer 实现的工程细节}

官方讲义明确提醒：课堂示例是慢速教学版。Assignment 1 要求学生实现更接近真实系统的 BPE tokenizer，并处理 special tokens、pre-tokenization 和速度优化。

\subsection{不要每次遍历所有 merges}

教学版 \texttt{encode} 对每个输入遍历所有 merges。如果 vocabulary 有几十万 token，merges 也可能有几十万条，这样的实现会非常慢。更好的思路是维护当前序列中的相邻 pair，只更新受一次 merge 影响的局部邻居。

\begin{table}[H]
\centering
\begin{tabular}{p{4cm}p{4.6cm}p{5.2cm}}
\toprule
\textbf{实现策略} & \textbf{优点} & \textbf{问题} \\
\midrule
遍历所有 merges & 概念简单，容易验证正确性 & 对长文本和大词表极慢 \\
维护 pair ranks & 只选择当前存在且 rank 最小的 pair & 数据结构复杂，需要处理局部更新 \\
预分词后逐段 BPE & 减少跨类别合并，符合 GPT 系 tokenizer 实践 & regex 和 special token 规则需要非常仔细 \\
\bottomrule
\end{tabular}
\caption{BPE encode 实现中的常见策略取舍。}
\end{table}

\subsection{Special tokens 不能被普通规则吞掉}

语言模型通常需要特殊 token，例如 \texttt{<|endoftext|>}、padding token、chat template 中的角色标记或工具调用标记。这些 token 不应该被普通 BPE 规则切碎，否则模型接口语义会被破坏。

\begin{warningbox}{特殊 token 是协议，不只是字符串}
\texttt{<|endoftext|>} 看起来像普通字符序列，但在模型协议里它有边界语义。如果 tokenizer 把它切成多个普通 token，训练和推理就可能无法正确表示文档结束、对话边界或工具调用结构。
\end{warningbox}

\subsection{Pre-tokenization：先切粗块，再做 BPE}

GPT-2 风格 tokenizer 会先用 regex 做 pre-tokenization，把文本拆成较粗的片段，再在每个片段内部做 byte-level BPE。这样可以减少一些不自然的跨类别合并，也让空格、数字、标点、单词边界的处理更可控。

Pre-tokenization 的重点不是追求“语言学上最正确的分词”，而是给 BPE 一个稳定的统计边界。对多语言、代码和数学文本来说，regex 规则会直接影响模型看到的 token 结构。

\subsection{Decode 的错误处理}

BPE 的 \texttt{decode} 通常先把 token IDs 映射回 bytes，再用 UTF-8 解码。这里可能遇到非法 byte 序列、未知 token ID 或特殊 token。教学版代码默认输入合法，真实系统必须明确错误策略：抛异常、替换为 replacement character，还是保留特殊 token 字面量。

\begin{knowledgebox}{Tokenizer 是模型 API 的一部分}
训练模型时的 tokenizer、推理服务的 tokenizer、评测脚本的 tokenizer 必须一致。换 tokenizer 等于换输入空间；即使模型权重不变，embedding 对应关系也会被破坏。因此，tokenizer 文件通常和 model checkpoint 一起发布。
\end{knowledgebox}

\subsection{本章小结}

Assignment 1 的 tokenizer 不只是算法练习。它要求学生面对真实系统问题：速度、special token 协议、pre-tokenization、错误处理和可复现性。一个慢速但正确的 BPE 能帮助理解概念，但一个可用的 tokenizer 必须把这些工程细节处理好。

\section{Tokenizer-free 梦想与现实约束}

讲义在 tokenization 部分也提到一个更长远的方向：tokenizer-free architectures。理想上，模型可以直接处理 bytes，不再需要人为设计 tokenizer。相关路线包括 ByT5、MEGABYTE、BLT、T-Free、H-Net 等。

\subsection{为什么直接建模 bytes 很诱人}

Byte-level 模型有几个理论优势：

\begin{itemize}
    \item 不需要处理未登录词，任意文本都可表示。
    \item 不把 tokenizer 作为固定前处理，减少手工归纳偏置。
    \item 对多语言、代码、emoji、噪声文本和新符号更统一。
    \item 可以避免 tokenizer 与模型能力之间的接口错配。
\end{itemize}

但它的成本也非常明显：byte 序列长，attention 和 KV cache 成本高；模型需要从更底层符号学习 chunk；在当前主流 Transformer 架构和硬件约束下，还没有在 frontier scale 成为主流。

\subsection{BPE 为什么仍然实用}

BPE 不是最终答案，却是今天很强的工程折中。它把常见片段压成较短序列，让 Transformer 在有限上下文和有限 FLOPs 下学习更高层模式；同时又保留 byte fallback，避免 word tokenizer 的 open vocabulary 问题。

\begin{importantbox}{tokenizer-free 与 BPE 的关系}
tokenizer-free 是一个重要研究方向，但当前课程从 BPE 开始是务实选择：它是许多现代语言模型实际采用的机制，也是理解资源效率、上下文长度和模型接口的最好入口之一。学懂 BPE，并不意味着相信 BPE 永远最优；它意味着理解当前系统为什么这样设计。
\end{importantbox}

\subsection{本章小结}

直接从 bytes 学习是优雅目标，但今天仍受序列长度和计算效率限制。BPE 的价值在于把覆盖性和压缩率结合起来，让主流 Transformer 在当前硬件约束下更可行。未来 tokenizer-free 模型如果成为主流，也仍然要回答同一个问题：如何在原始信号上形成高效的可学习 chunk。

\section{总结与延伸}

Lecture 1 的真正作用是建立一套学习语言模型的坐标系。它先指出研究者与底层技术脱节的问题，再承认前沿模型工业化带来的不可复现边界；随后用 mechanics、mindset、intuitions 区分哪些知识能在课堂规模迁移，哪些经验必须谨慎外推；最后把全课结构落到第一个具体模块 tokenization。

\subsection{核心要点提炼}

\begin{enumerate}
    \item CS336 的 from-scratch 哲学是为了获得因果理解，而不是为了复刻前沿模型。
    \item 前沿模型昂贵且闭源，因此课堂重点是可迁移 mechanics 和 resource mindset。
    \item Bitter Lesson 的正确读法是：可扩展算法和效率最重要，而不是算法不重要。
    \item 语言模型技术栈由历史积木累积而成：n-gram、LSTM、attention、Transformer、scaling laws、开放模型、agents。
    \item 五个课程单元围绕 efficiency 展开：basics、systems、scaling laws、data、alignment。
    \item Tokenizer 是模型的第一层接口，直接影响序列长度、词表大小、训练效率和生成可逆性。
    \item BPE 是一个实用 heuristic：从 bytes 出发，合并高频相邻 pair，在覆盖性和压缩率之间折中。
\end{enumerate}

\subsection{跨章节关联}

Tokenization 和 systems 看似相距很远，其实通过序列长度连接：token 数越多，attention、KV cache、decode latency 越贵。Tokenization 和 data 也相连：BPE merges 来自训练语料，语料分布会改变 token 统计。Tokenization 和 alignment 同样相关：chat template、special tokens、tool-call markers 都需要 tokenizer 协议稳定支持。

因此，第一讲的一个隐藏主题是：语言模型没有“纯局部模块”。一个 tokenizer 决策会影响 architecture；一个 architecture 决策会影响 systems；一个 systems 瓶颈会反过来限制 scaling recipe；一个数据决策会改变模型能力；一个 alignment 决策会改变推理负载。

\subsection{开放问题}

\begin{itemize}
    \item Tokenizer-free 模型能否在 frontier scale 上同时保持 byte-level 覆盖性和高效率？
    \item 如果 agentic 系统大量使用工具调用和结构化 trace，tokenizer 应该如何表示代码、JSON、终端输出和长上下文状态？
    \item Scaling laws 在后训练、agentic trajectories 和 inference-time compute 中是否仍然有同样强的预测性？
    \item 开放模型社区能否公开足够多的训练细节，让课程级复现与前沿经验之间的距离继续缩小？
\end{itemize}

\begin{importantbox}{最终 takeaway}
第一讲把 CS336 的学习方式压缩成一句话：在固定资源下，用可实现、可测量、可解释的方式训练更好的语言模型。Tokenization 是这条路的第一步，因为它决定模型从什么“原子”开始看世界；后续课程会继续沿着 efficiency 这条线，把架构、训练、系统、数据和对齐逐一拆开。
\end{importantbox}
