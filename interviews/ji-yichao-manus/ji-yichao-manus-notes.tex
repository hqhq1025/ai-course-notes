\section{引言}

2025 年初，Manus 以通用 Agent 身份引发全球关注，此后经历了跌宕起伏的一年。本访谈邀请了 Manus AI 联合创始人兼首席科学家季逸超（Pic），围绕创业经历、Manus 技术路线、Agent 架构设计、团队管理等主题展开了深度对话。

\begin{knowledgebox}{嘉宾简介}
季逸超，Manus AI Co-Founder \& Chief Scientist。高中时期在 App Store 发布 iOS 浏览器 MonkeySquare Browser（盟码浏览器），大学辍学创业，此后深耕 NLP 与知识图谱领域十余年，最终联合创办了 Manus。
\end{knowledgebox}


\section{创业历程：从浏览器到 NLP 再到 Agent}

\subsection{App Store 与第一桶金}

季逸超的创业起点是 2009 年 App Store 刚刚诞生的时代。作为中国第一代软件出海创业者，他在高中时期开发了第三方 iOS 浏览器 MonkeySquare Browser：

\begin{itemize}
    \item \textbf{商业模式}：最朴素的 Buy Copy 模式——每卖一份赚固定销售额
    \item \textbf{总收入}：约 30 万美金（高中生期间）
    \item \textbf{关键洞察}：App Store 的出现让个人开发者第一次有了全球化变现的途径
\end{itemize}

\begin{importantbox}{时代红利与平台迁移}
从桌面互联网到移动互联网，硬件媒介的变化创造了"蛮荒期"——巨头与个人开发者"众生平等"。但 AI 时代没有全新平台出现，大家的反应速度都一样快，不存在类似的蛮荒红利。
\end{importantbox}

\subsection{从浏览器到 NLP}

开发浏览器的过程中，季逸超萌生了"预测用户下一次点击"的需求（用于 Preloading 加速），这引导他进入了自然语言处理（NLP）领域。

2013 年 Word2Vec 论文的发布是关键转折点——第一次能将离散文本高效地转化为稠密向量，打开了"新世界的大门"。

\subsection{第二次创业：语义搜索与知识图谱}

受可穿戴设备（Apple Watch）的启发，季逸超看到了搜索交互革命的可能性——传统的"十个蓝色链接"在小屏幕和语音界面上不再适用，需要更紧凑的知识表示。

\begin{knowledgebox}{Open Information Extraction}
团队的技术突破是 Open IE（开放式信息提取）——不需要预定义实体类型或关系 schema，模型能自动从文本中提取三元组并构建知识图谱。这与传统 NER + Relation Extraction 需要白名单的方式截然不同。
\end{knowledgebox}

技术路线经历了多次迭代：Word2Vec → LSTM + Attention → Transformer/BERT，每次迭代都让前几年的积累归零。团队从零开始预训练了两个约 0.3B 的模型，并自研了基于 HNSW 算法的向量搜索系统。

\begin{warningbox}{创业教训：太早 = 先烈}
产品层面失败的三个原因：(1) 技术上低估了非技术壁垒（如数据源合作）；(2) 新的人机界面（可穿戴）没有按预期到来；(3) 商业模式从 2C 摇摆到 2B，团队基因不匹配。"早一步是先驱，早十步就是先烈。"
\end{warningbox}

\subsection{短暂的打工生涯}

创业结束后，季逸超在一家独角兽公司工作了一年半，主要任务是"打榜"——将用户需求转化为可量化的 benchmark，然后竞争排名以获取更多 GPU 资源。这段经历让他积累了在企业级 AI 产品中落地的实战经验。

GPT-3 的出现让他意识到危机，随后他在公司内部从零搭建了 LLM 业务线。

\subsection{本章小结}

从 App Store 到 NLP 到知识图谱再到 LLM，季逸超的每一次转型都由具体的产品需求牵引。"出海 + AI + 变现"的经验，最终汇聚成了创办 Manus 的基础。


\section{Manus 的诞生与定位}

\subsection{为什么做通用 Agent}

传统创业理念主张找 Niche Market 做垂直深耕。Manus 选择做通用 Agent 基于三个判断：

\begin{importantbox}{通用 Agent 的三层逻辑}
\begin{enumerate}
    \item \textbf{技术判断}：Manus 底层是通用模型 + 通用计算环境（虚拟机），走垂直反而是在上面加约束
    \item \textbf{产品判断}：先提供通用能力，观察用户集体行为模式（达尔文式观察），再做头部场景的定向优化
    \item \textbf{频次判断}：通用产品能覆盖"长尾场景"——对个体用户来说是高频刚需，但垂直产品不会为其专门服务
\end{enumerate}
\end{importantbox}

\subsection{为什么不做 AI 浏览器}

Manus 团队在 2024 年 4--10 月曾尝试做 AI Native 浏览器，但最终放弃。反思的核心问题：

\begin{enumerate}
    \item \textbf{共用电脑的尴尬}：AI 接管用户浏览器时，用户一滚屏就打破了 Agent 的 Observation，两人抢一台电脑的体验很差
    \item \textbf{短任务不值得}：简单几步点击就能完成的事，用户自己做比等 AI 推理更快
    \item \textbf{长任务的矛盾}：真正有价值的长任务需要持续运行，但本地浏览器关盖就休眠
    \item \textbf{缺乏差异化}：Chrome + Monica 插件就能做到同样的事
\end{enumerate}

\begin{warningbox}{浏览器迁移的历史规律}
人类历史上只有两次浏览器迁移（Netscape→IE, IE→Chrome），根本驱动力是\textbf{渠道分发能力}而非功能差异。AI 创业者不应低估这一壁垒。
\end{warningbox}

\subsection{Manus 的"Aha Moment"：长尾场景的惊喜}

Manus 的真正价值体现在长尾场景。一个典型案例：一位分子生物学家上传了实验仪器导出的小众数据格式——

\begin{enumerate}
    \item 其他产品完全不支持
    \item Manus 先去 Deep Research 了解这个格式
    \item 自行去 GitHub 下载开源项目来解析
    \item 完成后续数据分析
\end{enumerate}

\begin{knowledgebox}{Google 搜索的类比}
用户用 Google 而非 Bing 的原因不是头部查询的差异（两者差不多），而是 Google 能在"长尾 Query"中给出惊喜。Manus 追求的是同样的定位——在通用能力的基础上，让长尾场景的用户获得 Aha Moment。
\end{knowledgebox}

\subsection{本章小结}

Manus 选择了"通用 Agent + 长尾场景"的定位，避开了 AI 浏览器的坑，通过虚拟机沙盒提供隔离的执行环境，让 Agent 能够独立完成长时间任务。


\section{Agent 技术架构}

\subsection{Manus 的核心架构：模型 + 计算机}

\begin{importantbox}{Manus = 通用模型 + 虚拟机}
每个 Manus Session 背后都有一个\textbf{单独隔离的虚拟机沙盒}。虚拟机是图灵完备的——理论上能运行任何算法。底层的两个技术供给（通用模型 + 通用计算环境）本身就是通用的，走垂直只是加约束。
\end{importantbox}

\subsection{不做 Multi-Agent 的理由}

\begin{warningbox}{Multi-Agent 的陷阱}
季逸超明确反对当前的 Multi-Agent 范式。核心论点：
\begin{itemize}
    \item 人类的分工模式不应直接映射到 AI 系统——模型与人"一点都不像"
    \item 当前模型能力还不足以支撑可靠的 Agent 间协作
    \item 问题应该用更简洁的系统架构来解决，而非堆叠 Agent
\end{itemize}
"强行把模型或 Agent 与人的常用思维体系去对齐，这事是不对的。"
\end{warningbox}

\subsection{Coding 是 Manus 的灵魂}

虽然 Manus 不是 Coding Agent，但 Coding 是其核心能力的支柱：

\begin{itemize}
    \item Agent 通过\textbf{写代码}来执行大部分操作（生成网页、处理数据、调用工具）
    \item SWE-bench 是重要的内部评测指标
    \item 自定义 Linux 发行版中有大量命令行工具，模型需要基于文档学会使用
\end{itemize}

\subsection{Tool 设计哲学：减法优先}

\begin{knowledgebox}{克制的产品理念}
其他 Agent 公司在不断\textbf{增加} Tool 的数量，Manus 每月在思考\textbf{能删掉什么}。保持 Tool 精简有两个好处：(1) 降低模型的决策复杂度；(2) 通用 Tool（如 Code Execution）天然覆盖更多场景。
\end{knowledgebox}

\subsection{本章小结}

Manus 的技术架构以"通用模型 + 虚拟机沙盒"为核心，坚持单 Agent 架构，将 Coding 作为万能工具，通过减法设计保持系统简洁。


\section{数据飞轮与评估体系}

\subsection{两种数据飞轮}

\textbf{飞轮一：Self-Evolving（无参数更新）}

基于用户使用数据，将通用的 Failure Pattern（失败模式）和用户共识性知识注入系统——用的人越多，失败率越低，完成任务的轮次开销越小。

\textbf{飞轮二：用户反馈驱动迭代}

\begin{itemize}
    \item 收集用户最朴素的反馈（1--5 星评分）
    \item 发现 Benchmark 表现好的架构/模型，用户真实评分不一定高
    \item 用户关注点（如网站美观度、PPT 长宽比）与理想化 Benchmark 差异很大
\end{itemize}

\subsection{评估体系}

\begin{importantbox}{Manus 的评估维度}
\begin{enumerate}
    \item \textbf{Coding 能力}：SWE-bench 等编程基准
    \item \textbf{GUI 理解}：Browser Use 中的界面理解能力
    \item \textbf{广义 Tool Call}：不仅包括 Function Calling，还有命令行工具的使用
    \item \textbf{美学性}：难以自动化量化，依赖人工主观评估
    \item \textbf{错误自我意识}：模型能否发现并修复自己的错误
\end{enumerate}
\end{importantbox}

评估团队约 10 人，同时负责自动化评测系统搭建和人工主观评估。

\begin{warningbox}{Agent 评估的根本困境}
自动化 Evaluation 在 Agent 场景下仍有很大改善空间。比如"网站是否好用"无法通过 Reward Model 自动判断。Manus 的做法：Agent 每做完一个网页，自己用浏览器玩一圈，再检查数据库记录是否对上——用通用能力做自我验证。
\end{warningbox}

\subsection{本章小结}

Manus 构建了无参数更新的 Self-Evolving 机制和基于用户反馈的迭代闭环。评估体系结合了自动化 Benchmark 和人工主观评估，尤其重视那些难以量化的维度。


\section{团队管理与决策哲学}

\subsection{GPA 决策框架}

Manus 团队使用 GPA 框架做决策：

\begin{itemize}
    \item \textbf{G（Goal）}：目标设定——CEO 专制拍板
    \item \textbf{P（Priority）}：优先级排序——专制 + 民主混合（可以讨论，但有人最终拍板）
    \item \textbf{A（Alternative）}：可选方案——充分民主（方案数量比质量更重要，避免 Action Space 本身有缺陷）
\end{itemize}

\subsection{核心理念：行动优先}

\begin{importantbox}{"与其悬而未决，不如赶紧试试"}
在新领域中：(1) 过往经验不一定有用；(2) 想更多不会产生新信息——你仍然基于模型内部的参数化知识，没有做 RAG；(3) 不如先执行，拿到 Reward 再迭代。
\end{importantbox}

\subsection{"不做什么"比"做什么"更重要}

AI 时代创业公司的产能被大幅解放，眼中的机会看似很多。Manus 的核心纪律是每天回答"不做什么"——保持产品克制。

\subsection{本章小结}

Manus 的团队管理注重决策效率（GPA 框架）、行动速度（先试再说）和产品克制（不做什么），避免在机会泛滥的 AI 时代陷入多线作战的陷阱。


\section{行业观察与前瞻}

\subsection{模型与 Agent 的关系}

季逸超反对"不训模型做 Agent 就是逆向工程"的说法。他的论点是：模型\textbf{无法 internalize 环境}，因此 Agent 层面的创新永远有价值。

同时他认为：

\begin{itemize}
    \item 很多能力最终会收敛到所有产品中（如 Deep Research）
    \item 真正的竞争力来自"长尾 Query 的惊喜"
    \item Agent 的差异化在于系统设计，而非单一模型能力
\end{itemize}

\subsection{多模态的优先级}

\begin{knowledgebox}{输入 vs 输出}
Manus 更重视多模态\textbf{输入}而非输出。在 Agent 场景中，一个重要观察是：当前很多多模态模型在处理 Tool Result 中返回的图片时，质量会下降——这块的训练重视程度还不够。
\end{knowledgebox}

\subsection{对行业领袖的评价}

季逸超对几位行业人物的评价：
\begin{itemize}
    \item \textbf{姚顺雨}：观点高度认同，尤其是"大厂抄小公司"和"下半场"的概念
    \item \textbf{杨植麟}："只要问题能被定义，问题一定能解决"——这是做事的信心来源
    \item 从狭义技术角度，最佩服的是 Demis Hassabis
\end{itemize}

\subsection{本章小结}

Agent 层面的创新有持久价值，因为环境无法被模型内化。竞争差异化来自系统设计和长尾场景的覆盖，而非堆砌模型能力。


\section{总结与延伸}

这场访谈覆盖了 Manus 从创业动机到技术实现再到行业前瞻的全貌。核心洞见可以归纳为：

\begin{enumerate}
    \item \textbf{产品定位}：通用 Agent 的价值在长尾场景——每个人的工作都是独特的
    \item \textbf{技术架构}：模型 + 虚拟机沙盒，单 Agent，Coding 是灵魂
    \item \textbf{设计哲学}：减法优先，克制比扩张更重要
    \item \textbf{评估体系}：自动化 Benchmark + 人工主观评估 + 用户反馈
    \item \textbf{决策文化}：行动优先，先执行再迭代
    \item \textbf{行业判断}：Agent 创新有持久价值，模型无法 internalize 环境
\end{enumerate}
