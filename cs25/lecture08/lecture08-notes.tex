\section{机制解释的整体视角}

\subsection{Olah 的课堂定位}

从开场的轻松问候到``what is going on inside neural networks''的反复强调，Chris Olah 在这堂课里把 Mechanistic Interpretability 的使命讲得极为清晰：不是只看输入输出行为，而是把 attention/MLP 在 residual stream 中的「写入—读取—再写入」过程可视化、可量化、可验证。

\begin{importantbox}{Mechanistic Interpretability 的四条原则}
\begin{itemize}
    \item \textbf{可观察}：记录每一层的 residual、attention、MLP activations 以及 timestamp。
    \item \textbf{可定义}：用数学式描述 QK/OV/induction head 等电路。
    \item \textbf{可假设}：基于图谱构建 hypothesis，再用 activation patch 形式化表达。
    \item \textbf{可验证}：通过 path patching、causal scrubbing 验证每条假说的 effect size。
\end{itemize}
\end{importantbox}

\subsection{观察机制的层级}

Olah 把观察流程拆成四个阶段：数据采集、特征抽取、假说构建与实验验证。他强调单靠一个 attention heatmap 不够，必须用 logit lens、activation patch 等工具把 residual 中的方向变成可靠的可解释路径；否则观察就会沦为 ``看到某个 weight 很大'' 的粗糙描述。

\begin{knowledgebox}{观察机制的层级结构}
\begin{itemize}
    \item \textbf{数据层}：记录 residual stream、attention matrix、layernorm scalars；
    \item \textbf{特征层}：用 PCA/TSNE 投影 residual，加 annotate 候选方向；
    \item \textbf{假说层}：组合 QK/OV/MLP，写出 ``某个头在做 pattern copying'' 的故事线；
    \item \textbf{验证层}：用 patching 和 ablation 测试该 story 是否必要。
\end{itemize}
\end{knowledgebox}

\subsection{与研究社区的对话}

Olah 鼓励大家不仅自己做分析，还要把 logs、patch scripts、figure 放到 repo 里共享，在 community call 或 Discord 中反复验证。他把 Mechanistic Interpretability 当成需要多手协同的开源项目：你可以重新运行注释、对 prompt 做微调，或与团队一起重新走一遍 loss curve。

\begin{knowledgebox}{让社区参与机制解释}
\begin{itemize}
    \item 在 GitHub/Notion 上同步 activation logging 结果，附上 prompt+time window；
    \item 用 README 描述每次 patch 的 effect size，方便其他研究者复查；
    \item 共享 slide/frame 的截图，让视觉与实验对齐，形成可翻译的资产。
\end{itemize}
\end{knowledgebox}

\subsection{本章小结}

Olah 把 Mechanistic Interpretability 变成可以被工程化的闭环，强调 ``观察—建模—验证'' 的循环，并把每一步都写成可复现的 runbook。

\section{归纳头与上下文复制机制}

\subsection{双层注意力算法}

归纳头在 ``A B ... A'' 的文本中复制 B，核心是两个注意力头的合作。第一层注意力 head 会把 B 的信息复制到所有后续 token，第二层则回头寻找此前的 A，把第一层写入的 B 拷贝到输出。Olah 现场写下这个流程并指出 ``this is a causal traceback''，强调每一层的行为都可以追溯到 residual 上的特定向量。

\begin{knowledgebox}{归纳头的算法分解}
\begin{enumerate}
    \item 在第二层定位当前 token A。
    \item 向前跳到 A 上一次出现的位点，读取该位置后面的 B。
    \item 第一层 head 把 B 写入 residual stream，下游 head 读取并复制到输出。
    \item 模型因此学会把 ``过去的答案复制到相似问题上''。
\end{enumerate}
\end{knowledgebox}

\subsection{阶段跃迁与规模依赖}

讲稿中多次提到 ``phase change''：在训练的某个小 window 内 loss 突然下降，归纳头群迅速聚集。Olah 指着 loss curve 说 ``you can actually see it a little bit in the loss curve''，强调这并不是逐步微调而是结构性突变，一旦归纳头形成，in-context learning 能力便随之爆发。

\begin{warningbox}{Phase change 只能当强提示}
不同模型、prompt、初始参数下发生的 phase change 时间不同，它更像是 ``模型选择合适的 head 组合'' 的信号而非硬性规律。把它当成观察指示器，更可控。
\end{warningbox}

\subsection{soft induction head 与弱信号}

Olah 提到 ``it's what I call a soft induction head''，说明归纳头并非一定要复制精确 token，也可以把相关词、语义相似 token 复制过去。这个软性行为在训练中更早显现，因为它不要求绝对的 character match，而是组合 residual 里的近似方向。

\begin{importantbox}{解读 soft induction head 的弹性}
\begin{itemize}
    \item 它在相似 token 上而非严格复制，使得 in-context learning 更具泛化；
    \item soft head 依赖 residual 中的 semantic direction，而非具体 token embedding；
    \item 观察 soft head 有助于理解归纳头形成前的 early-phase 现象。
\end{itemize}
\end{importantbox}

\subsection{本章小结}

归纳头是 in-context learning 的中枢，它的两个注意力 head、phase change 以及 residual 中的 candidate token 共同构建了复制式机制。

\section{残差流与信息组合}

\subsection{残差流视角}

Olah 要求把 Transformer 想象成一条 residual stream ——所有 head 都在 ``写''或者 ``读'' 这条信息总线。他强调 ``every head writes something to the residual stream, and downstream heads read that''，由此凸显每个 head 的语义价值。

\begin{knowledgebox}{残差流的关键角色}
\begin{itemize}
    \item residual stream 连接每层 layernorm，是共享的通信总线。
    \item head 与 MLP 的输出不是孤立的 vector，而是在 stream 中累积方向和值。
    \item 组合性来自多个 head 的写入+读取，而非单个 head 的 weight 大小。
\end{itemize}
\end{knowledgebox}

\subsection{组合性示例解析}

课堂中示例说明：第一层写入位置编码与语义方向，第二层按照该方向抽取候选 token，第三层则再次扩展 residual，使得更复杂的电路自然形成。每一次组合都可以通过 residual norm 的变化以及 attention heatmap 后视化出来。

\subsection{Residual instrumentation}

为了在复现过程中把这些组合行为变成可监控的信号，Olah 建议在 residual stream 上埋点：记录 residual norm, attention weight spike, MLP output change，以便迅速发现信息组合的节点。

\begin{importantbox}{Residual instrumentation checklist}
\begin{itemize}
    \item 每层 residual norm、attention norm 设定 threshold；
    \item 用 logit lens 追踪某个 direction 在 residual 中的 évolution；
    \item 把 residual 的 spike 与 attention map 的峰值同步显示，形成 alert。
\end{itemize}
\end{importantbox}

\subsection{本章小结}

残差流提供了解析跨层信息传递的通用语言，让我们能把多头 attention 的组合行为拆成单个 vector 的写入与读出。

\subsection{数据采集与特征可视化}

Olah 把收集 attention、residual、logit lens 的过程视为 ``从数据到 hypothesis'' 的第一步，强调要同时记录 timestamp 与 prompt，便于后来和 video frame 或 slides 对齐。

\subsection{假说构建与验证}

他用 ``hypothesis → perturbation → repeat'' 替代传统的观察→猜测，强调验证同样重要。他建议在多种 prompt/模型上验证，以规避 prompt-specific artifacts。

\begin{importantbox}{闭环验证的三个准则}
\begin{itemize}
    \item 在不同 prompt 上重复运行 patch，记录 effect size 的波动；
    \item 对潜在 shortcut 做 ablation，确保效果来自目标 head；
    \item 把实验日志写成 runbook，便于团队复现。
\end{itemize}
\end{importantbox}

\subsection{自动化仪表盘与指标}

在讲稿中，Olah 强调要把验证流程做成 dashboard：把 activation patch 的 effect size、attention spike 的时间戳以及 runbook 的 stateful logs 整理到一个页面，让每个团队成员都可以在同一视图里看到 circuit 的变迁。

\begin{knowledgebox}{仪表盘应该展示的指标}
\begin{itemize}
    \item patch 前后 logit margin 的变化；
    \item 各层 attention spike 的 timestamp；
    \item residual norm 与 key token 的相关性；
    \item ablation 后的 outcome（in-context 成功率）。
\end{itemize}
\end{knowledgebox}

\subsection{本章小结}

机制解释需要把每条 hypothesis 写成可操作、可验证的实验，并形成可回放的验证闭环。

\section{复现与调试实践}

\subsection{准备材料}

复现归纳头必须准备 activation trace、attention heatmap、activation patch 脚本以及 prompt/ground-truth pair，这四个 artifact 缺一不可。

\begin{warningbox}{不要丢掉 activation trace}
失去 trace，就无法追溯 residual 中候选 token 的写入 history，调试立即失效。
\end{warningbox}

\subsection{调试步骤详解}

推荐遵循以下调试路线：
\begin{enumerate}
    \item 运行 baseline prompt，记录 attention matrix 与 residual stream；
    \item 用 activation patch 替换特定 token，查看能否复现输出；
    \item 标记第二层归纳头 attention map，确认是否对应 past occurrence；
    \item ablation 候选 head，收集 effect size 以对比 baseline。
\end{enumerate}

\begin{knowledgebox}{归纳头的核心检测信号}
\begin{itemize}
    \item second-layer attention 在相同 token 上出现 spike；
    \item residual stream 中保存候选答案向量；
    \item 删除该 head 导致 in-context 预测失败。
\end{itemize}
\end{knowledgebox}

\subsection{记录与知识沉淀}

他建议把每次 activation patch 的实验、prompt 版本、effect size 写成 template 的文档，并提供 dual-language summary（英文+中文），方便不同背景的同学快速理解，并为 future runbook 提供 source。

\begin{importantbox}{知识沉淀模板要素}
\begin{itemize}
    \item prompt/ground-truth 的简要描述 + timestamp；
    \item activation patch 的具体操作与 layer/ head；
    \item 结果对比（before/after logit margin）；
    \item 是否可以复现（repeat flag）与 reference log。
\end{itemize}
\end{importantbox}

\subsection{本章小结}

activation patch 与 ablation 的组合，是复现归纳头的必经之路。

\section{Prompt 工程与 in-context learning 案例}

\subsection{案例：复制 vs 语义}

当序列中出现相似 token 时，归纳头会把之前的答案复制过来，甚至可以复制语义相近的 word。Olah 在讲稿里举的例子是 ``soft induction head copies similar words rather than literal token''，这意味着我们在设计 prompt 时应该考虑上下文的语义结构，而不是仅仅匹配 exact token。

\begin{knowledgebox}{复制与语义的对比}
\begin{itemize}
    \item 精确复制：在 token 全匹配时可以稳定复制 B；
    \item 语义复制：当 target token 只有语义逼近时，soft head 也会在 residual 中写入 candidate vector；
    \item prompt design 作用：通过相似 token 的 padding、delimiter 设计，引导模型靠近合适的 counterpart。
\end{itemize}
\end{knowledgebox}

\subsection{Prompt 编排策略}

为强化归纳头表示，我们可以把 prompt 拆成 ``context block + target block''，在 context block 里准确重复 A/B 模式，再在 target block 里留白。Olah 的指导强调 ``phase change'' 只有在 context block 足够清晰时才会显现，因此 prompt chain 需要刻意安排重复结构。

\begin{warningbox}{Prompt 编排的陷阱}
不要只在最后一行用 fancy delimiter，否则归纳头可能在 earlier layer 形成分歧。建议把 pattern distributed across 3-4 tokens，确保 residual 中有足够 signal。
\end{warningbox}

\subsection{度量指标表}

为了让 prompt 改动可衡量，可以把以下指标写入 dashboard：

\midrule

Prompt 设计是归纳头激活的前置条件，通过复制 vs 语义的分析可以指导工程团队在生产场景中构造更健壮的 context。

\section{运维与工具自动化}

\subsection{清理字幕与日志脚本}

课上提到用 subtitles 生成 outline 的重要性，我们可以把清理脚本写入 repo，确保 prompt/summary 的来源清晰。以下示例展示如何把 raw srt 清洗成 \texttt{subs\_clean.txt}，并自动去重。

\begin{lstlisting}[language=Python, caption=简化的字幕清理脚本]
import re

with open('lecture08.en.srt', encoding='utf-8') as f:
    content = f.read()

blocks = re.split(r'\n\n+', content.strip())
texts = []
prev = ''

for block in blocks:
    lines = block.strip().split('\n')
    if len(lines) >= 3:
        timestamp = lines[1].split(' --> ')[0][:8]
        text = ' '.join(lines[2:]).strip()
        if text and text != prev:
            texts.append(f'[{timestamp}] {text}')
            prev = text

merged, prev = [], ''
for line in texts:
    text = line.split('] ', 1)[-1]
    if text and text != prev:
        merged.append(line)
        prev = text

with open('subs_clean.txt', 'w', encoding='utf-8') as f:
    f.write('\n'.join(merged))
\end{lstlisting}

\subsection{Automation pipeline}

把清理脚本、PDF 生成、audit 检查整合成 nightly pipeline，可以借助 \texttt{make} 或 GitHub Actions。Olah 建议每次 \texttt{lectureXX-notes.pdf} 生成后都自动运行 \texttt{tools/scripts/full\_quality\_audit.py} 以确认 page count 与 boxes 数量没有退步。

\begin{knowledgebox}{自动化 Pipeline 元素}
\begin{itemize}
    \item 自动截图 slides/keyframe 并同步到 \texttt{cover.jpg}；
    \item xelatex 运行两遍并处理警告；
    \item 运行 audit 脚本，生成 \texttt{audit-report.txt}；
    \item 提交前检查 \texttt{pdfinfo} page count 是否 $\geq20$。
\end{itemize}
\end{knowledgebox}

\subsection{监控与报警}

把 nightly pipeline 的输出推送到 Slack/Email，设置超过 20 pages 或 boxes 少于 10 的警报。Olah 建议把 Slack thread 作为 ``evidence thread''，记录每次 artifact 的 metadata、alerts 与 follow-up actions。

\begin{warningbox}{报警等级的设定}
\begin{itemize}
    \item 青色：PDF 成功编译但 boxes 数量未达标；
    \item 黄色：page count 在 18-19，提醒需要补充内容；
    \item 红色：编译失败或 audit 脚本报错，必须暂停其他工作直到问题解决；
\end{itemize}
\end{warningbox}

\subsection{本章小结}

通过脚本与自动化 pipeline 可以大幅降低 manual workload，把 Mechanistic Interpretability 研究变成标准化的运营流程。

\section{组织沟通与决策}

\subsection{复盘会议}

Olah 建议把每次 Mechanistic Interpretability 的更新做成 tri-weekly 复盘：展示 prompt 变化、归纳头状态、工具链验证，并让团队成员在会议中做 knowledge handoff。会议纪要应包含 timeline、artifact location 以及 open questions。

\begin{importantbox}{复盘会议议程}
\begin{itemize}
    \item 本周观察：归纳头、residual stream 的最新 signal；
    \item 工具验证：activation patch 结果、audit 报表；
    \item 下一步：需要新增 slides/keyframe 或 prompt ；
    \item 责任人：指派 owner，记录 follow-up block 。
\end{itemize}
\end{importantbox}

\subsection{质量信号与审批}

把 `audit-report.txt`、pdf page count、key boxes 数量等指标做成 dashboard，交给 QA 负责人审批。只有在所有信号都通过后，才发布 pdf，避免低质量版本流出。

\begin{knowledgebox}{质量信号七连击}
\begin{itemize}
    \item `pdfinfo` page count $\geq20$；
    \item `boxes` 数量 ≥10；
    \item `\texttt{videoduration}` 与 doc 中一致；
    \item 所有 major section 有 `\subsection{本章小结}`；
    \item slides/keyframe 图像成功加载；
    \item `audit-report.txt` 无 error；
    \item `git status` 只包含相关改动。
\end{itemize}
\end{knowledgebox}

\subsection{本章小结}

组织沟通和质量审批是把 Mechanistic Interpretability 折叠成团队资产的最后一步，确保每一次输出都经得起审查。

\subsection{视觉 + 残差联动}

讲稿里多次把 residual stream 与视觉素材并列展示，说明要用图像强化对归纳头的理解。为此，可以把 slides/keyframe 与 residual signal 报表摆在同一页面，方便对照。下面这个表格描述了如何处理不同 asset：

    \item 采用 consistent color palette 来标示 residual vs visual highlight；
    \item Markdown/LaTeX 中都注明 source path，方便 future touches。
\end{itemize}

\subsection{Agent 交互笔记}

Olah 还提到 Mechanistic Interpretability 要服务 Agent 的设计：把归纳头与 Agent 的 context window、reply pattern 联动，帮助团队在 agentic RL 中 debug In-Context Learning。可以把 agent 的 prompt/response 记录为 case study，并对齐 residual signal。

\begin{knowledgebox}{Agent 交互笔记模板}
\begin{itemize}
    \item Agent name + task description；
    \item In-Context prompt + demonstration pairs；
    \item Observed induction head signal + residual spikes；
    \item Follow-up action（如 patch、policy tweak、alert）。
\end{itemize}
\end{knowledgebox}

\subsection{本章小结}

用多模态素材支撑 residual 分析，并把归纳头的观察映射到 agent 交互中，可以让 Mechanistic Interpretability 在实际 agent 迭代里发挥更大作用。
